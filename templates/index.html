Overview

Knock is a lightweight web application for managing and waking compute nodes using Wake-On-LAN magic packets. Add a node once by MAC address and wake it from any browser on your network.

  • Add, edit, and delete nodes via the web UI or REST API
  • Sends a standard 102-byte magic packet over UDP broadcast — no sudo required
  • Nodes persist in a local JSON file; no database needed
  • Dark mode, mobile-responsive interface

How Wake-On-LAN Works

When you click Wake, Knock sends a 102-byte magic packet over UDP broadcast to port 9. The target machine's NIC listens for this packet even while the system is powered off (as long as it receives standby power).

Packet structure

Bytes  0–5   │ FF FF FF FF FF FF          (6-byte sync stream)
Bytes  6–11  │ Target MAC address         (first repetition)
Bytes 12–17  │ Target MAC address         (second repetition)
  …          │   …
Bytes 96–101 │ Target MAC address         (sixteenth repetition)
─────────────┴──────────────────────────
Total: 102 bytes, sent via UDP to 255.255.255.255:9

Requirements on the target machine

  • Wake-On-LAN enabled in BIOS/UEFI
  • WoL enabled on the NIC — on Linux: ethtool -s eth0 wol g
  • The machine must be on the same Layer 2 segment as Knock, or the network must be configured to forward the packet (see Cross-Subnet below)

MAC Address Formats

Knock accepts any of the following formats and normalises them to xx:xx:xx:xx:xx:xx on save:

00:11:22:33:44:55      ← colon-separated (most common)
00-11-22-33-44-55      ← hyphen-separated
001122334455           ← plain hex string

Find a machine's MAC address with:

# Linux
ip link show
# macOS
ifconfig | grep ether
# Windows
ipconfig /all

Waking Nodes Across Subnets

UDP broadcast (255.255.255.255) is confined to the local subnet — routers drop it by default. If Knock and the target are on different network segments, one of the following approaches is needed.

1

Run Knock on the same subnet

The simplest fix. Deploy Knock on any machine that shares a Layer 2 segment with the targets. Broadcast stays local, no router changes required.

[Browser] ──HTTP──► [Knock @ 192.168.10.x] ──UDP broadcast──► [Target @ 192.168.10.y]
2

Directed broadcast (router config)

Send the packet to the subnet's broadcast address (e.g. 192.168.20.255 for a /24) rather than 255.255.255.255. Many routers can be configured to forward directed broadcasts.

On a Cisco IOS router, enable it per interface:

interface GigabitEthernet0/1
 ip directed-broadcast
Directed broadcast forwarding is disabled by default because it can amplify DDoS attacks (Smurf attack). Only enable it on trusted internal interfaces.
3

UDP relay / WoL proxy

Run a small agent on the remote subnet that listens for a forwarded unicast packet and re-broadcasts it locally as a magic packet. 

[Knock] ──UDP unicast──► [relay @ 192.168.20.x] ──broadcast──► [Target @ 192.168.20.y]

Relay options:

  • wol-relay — lightweight daemon, listens on a unicast port and rebroadcasts
  • Home Assistant — built-in WoL integration, place an instance per subnet
  • OpenWrt etherwake — trigger via SSH from the Knock host
4

SSH + remote etherwake

If you have SSH access to any machine on the remote subnet, trigger wakeonlan or etherwake over SSH — no relay daemon needed. 

ssh user@remote-host "wakeonlan 00:11:22:33:44:55"
# or
ssh user@remote-host "etherwake -i eth0 00:11:22:33:44:55"
5

VLAN-aware switch WoL passthrough

Some managed switches (Ubiquiti, Cisco SG series, HP ProCurve) can pass magic packets between VLANs at Layer 2. Configured on the switch itself — no code changes needed. Check your switch docs for "WoL across VLANs" or "magic packet forwarding".

6

VPN / site-to-site tunnel

If the remote subnet is connected via a site-to-site VPN (WireGuard, IPsec, etc.), the VPN gateway on the remote side is a machine on that subnet. Install Knock there, or use the SSH method through the tunnel.

API Reference

All endpoints return JSON. The base URL is http://<host>:5000.

MethodEndpointDescription
GET/api/statusServer status and node count
GET/api/nodesList all nodes
GET/api/nodes?enabled_only=trueList enabled nodes only
GET/api/nodes/<id>Get a specific node
POST/api/nodesCreate a node
PUT/api/nodes/<id>Update a node
DELETE/api/nodes/<id>Delete a node
POST/api/nodes/<id>/wakeSend WoL magic packet
GET/api/interfacesList network interfaces

Example: create a node

curl -X POST http://localhost:5000/api/nodes \
  -H "Content-Type: application/json" \
  -d '{"mac_address":"00:11:22:33:44:55","hostname":"server-01","enabled":true}'

Example: wake a node

curl -X POST http://localhost:5000/api/nodes/0/wake

Troubleshooting

Node does not wake up

  1. Confirm WoL is enabled in the target machine's BIOS/UEFI.
  2. Confirm WoL is enabled on the NIC: 
    ethtool eth0 | grep Wake   # check current state
ethtool -s eth0 wol g      # enable magic packet wake
  3. Verify the MAC address stored in Knock matches the target's NIC MAC exactly.
  4. Confirm Knock and the target are on the same subnet, or that one of the cross-subnet approaches above is configured.

Port 5000 already in use

WOL_PORT=8080 python wol_server.py
# or with uv:
WOL_PORT=8080 uv run wol-server

Server status shows Disconnected

  • Check the server is running: ps aux | grep wol_server
  • Verify port 5000 is not blocked by a firewall
  • In Docker on macOS/Windows, network_mode: host is not supported — run the server directly with Python instead

Docker container can't send broadcast

The production docker-compose.yml uses network_mode: host so the container shares the host's network stack and UDP broadcast reaches the LAN. This only works on Linux hosts.
